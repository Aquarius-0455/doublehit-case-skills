# doublehit-case-skill 测试用例生成工具 (快捷唤醒: 双击666)

本工具箱用于解析输入的需求文本，提取业务流转和判定节点，生成完整的测试用例。
**核心原则：优先对需求文档所有明文功能点、判定分支与交互动作进行完整覆盖。在此基础上，进行异常分支延伸及等价类合并。**

> 在开始任何逻辑分析前，**必须先执行 `ls -R` 或 `dir` 指令**，动态确认当前工作目录以及脚本 `skills/doublehit-case-skill/scripts/case_exporter.py` 的具体物理路径。
> **严禁硬编码绝对路径**，必须根据自检结果动态拼接后续的写入与导出指令。

## 1. 收到需求后的第一步：输入识别

> 拿到任何输入，**先判断类型，再决定怎么处理**，不允许立即生成。

| 输入特征 | 判定 | 处理方式 |
|---|---|---|
| 功能列表、字段定义、状态码表、业务规则 | 需求文档 | 直接进入分析 → 生成 |
| Story 编号（如 DTS-/BI-）+ 结构化描述 | Story | 直接进入分析 → 生成 |
| 口语化描述（如"帮我写一下登录的用例"） | 随意输入 | 先整理为结构化需求，确认后生成 |
| 仅一句话 / 缺少字段/规则/边界描述 | 模糊输入 | **追问**：场景是什么？有哪些字段？规则是什么？ |

**删除线处理**：需求文本中 `~~内容~~` 包裹的内容代表已取消的功能/字段，
**必须扫描并列出所有删除线内容，严禁为其生成任何用例。**

## 1. 设计方法选择决策

> 分析需求类型后，从下表选定方法，**不允许无脑套固定方法组合**。

| 需求特征 | 优先方法 |
|---|---|
| 输入字段有范围限制（数值、日期、金额） | 边界值分析 + 等价类 |
| 多个输入条件共同决定一个输出 | **因果图** → 转判定表 |
| 多条件独立且全排列用例爆炸 | 正交实验法 |
| 业务流程有明确状态变化（订单/审批状态机） | 状态迁移法 |
| 用户完整操作路径型业务 | 场景法 |
| 历史 Bug 集中区域 / 经验判断高风险点 | 错误猜测法 |
| 多业务规则组合产生不同结论 | 判定表法 |
| 输入域可分为合法/非法/边界类别 | 等价类划分 |

## 1.5 需求原子节点提取 (Atomic Node Extraction) - 必做

> 在进入设计前，必须先执行此步，将冗长的需求文本拆解为一个个可验证的“原子约束点”。

**提取规则：**
- **名词提取**：提取所有涉及的业务参数（如：重量、体积、目的地、Request No、审计快照）。
- **动词捕捉**：捕捉所有转折与判定（如：一旦满足...则、如果是...则跳过、必须在...之前执行）。
- **场景扫描**：列出所有定义的业务路径（如：Path A/B/C、流转分支1/2/3）。
- **时机锁定**：记录所有对执行顺序的要求（如：生成后下发前、勾选后置灰）。

**输出要求：** 在“分析预览”中以列表形式展示提取出的核心原子约束，确保后续设计能“对号入座”。

## 2. 测试设计核心规则

### A. 三层延伸模型
- **L1 业务层**：正常路径，正向业务闭环。
- **L2 稳定性层**：接口超时、异常跳过、数据极值、反向/非法输入拦截。
- **L3 工程安全层**：并发冲突、权限越权、审计完整性、数据一致性。

### B. 测试方向优先级：功能绝对为主，非功能抽样为辅
用例集的构建必须贯彻“主次分明”的思想：
- **功能测试（核心主体）**：倾注绝大部分精力，对所有的业务分支、决策矩阵、正常异常路径做详尽的全逻辑覆盖。
- **接口/性能/安全/易用性（辅料点缀）**：绝不全量铺开，仅针对架构上最容易脱轨的薄弱环节（如外网掉线、表单高频强切、越权调用），**只抽取最典型的两三条代表性场景即可，点到即止，切忌喧宾夺主**。

#### 功能测试
覆盖核心业务分支和逻辑决策矩阵；验证每条规则的正常路径和异常路径。

#### 接口测试
- **请求层**：HTTP 方法、必填参数缺失、参数类型错误、参数超长。
- **鉴权层**：无 Token、Token 过期、权限不足、多租户越权。
- **响应层**：状态码（200/400/401/403/404/500）、响应结构、字段类型。
- **异常层**：超时回退、重复请求幂等性、并发数据一致性。

#### 性能测试
响应时间 P99 / 吞吐量 TPS / 压力下降级行为 / 长时运行资源泄漏。

#### 安全测试
敏感操作二次确认 / 前端参数篡改防御 / 越权拦截 / XSS、SQL 注入、CSRF。

#### 易用性测试
- 非法输入时错误提示是否清晰、指向具体字段。
- 核心任务步骤数是否合理，无多余跳转。
- 字段达到长度上限时是否有实时提示而非提交后报错。
- 列表为空时是否有引导文案。
- 提交失败后表单内容是否保留。

#### 接口契约测试
Payload 结构 / 必填项缺失 400 响应及错误描述准确性 / 异步失败 Fallback 验证。

### C. 设计方法详解

**等价类**：分有效/无效类，每类取 1 个代表值。两类触发同一代码路径 → 只保留边界值。

**边界值**：取点 `[min-1, min, min+1, max-1, max, max+1]`，结合等价类去重。

**因果图**：列出所有"原因"（输入条件）和"结果"（输出动作）→ 画逻辑关系（AND/OR/NOT）→ 转化为判定表 → 合并相同结果的列去冗余。

**判定表**：枚举条件组合 → 合并触发相同动作的列 → 每列一条用例。

**状态迁移**：画状态转换图 `[当前状态]+[事件]→[目标状态]`，覆盖合法迁移 + 非法迁移尝试。

**错误猜测**：聚焦历史 Bug 密集模块、第三方集成点、写后读一致性、极端值组合场景。

**正交实验**：使用正交表选最小用例集，保证任意两因子的取值组合至少出现一次。

### D. 执行红线（全覆盖 + 典型代表延伸 + 去重合并）
- **第一步：显性需求 100% 颗粒化覆盖**：逐句解析需求文档。任何条件判定、流转走向、按键交互、**乃至审计日志的字段要求**，都必须落到具体的测试用例中。
- **第二步：分流路径饱和覆盖**：文档中定义的每一个 Path（路径A/B/C）或流转决策分支，必须拥有至少一条独立的 L1/L2 用例，严禁在主流程用例中“顺带测试”分支逻辑。
- **第三步：非功能性边缘延伸**：在一经跑通全量功能的基础盘上，额外补充少量、且最具代表性的薄弱环节（如：唯一一条涉及 API 断网的 Fallback 场景，或者唯一一条防重敲的 UI 互斥防线），只摘取精华。
- **第四步：等价类合并与去重**：
  - 同段代码处理逻辑通过等价类合并去除重复操作，同一条 IF 路径不留重复条目。
- **第五步：文风要求：专业严谨、数据具象（严禁模糊谓词）**：
  - **强制具象化**：严禁使用“输入大金额”、“选择一个物流”等描述。必须使用“输入 99999 USD”、“选择 Carrier=DHL”等具体数据，确保测试执行的唯一性。
  - 一律使用“标准系统名词 + 具象数据条件”的严密格式。
- **预期结果三要素进化版**：
  - **[UI反馈]**：必须包含弹窗内容对比、按钮状态（置灰/激活）变化。
  - **[状态回写]**：必须指明后端具体的**字段名及状态值**映射。
  - **[审计记录]**：必须完整对应需求中对审计链路的描述（如：初始方案 -> 推荐方案 -> 手动切换 -> 最终方案）。
- **步骤原子化**：每步对应明确具体的操作点击或数据投喂动作。

## 3. 用例生成规范

### 第一步：分析预览
1. 扫描删除线内容 → 列出已取消项，不为其生成用例。
2. 判断输入类型 → 选定设计方法（见第 1 节）。
3. 输出 Markdown 决策矩阵：`条件组合 → 预期动作 → L1/L2/L3 层级`。
4. 配比确认：正向 40% / 反向异常 40% / 边界稳定性 20%。

### 第二步：生成前自我稽核 (AI Self-Check)
在正式产出 JSON 前，必须在内部大脑/草稿中进行“三道红线”拦截校验：
1. **标题雷达**：盘点梳理出的标题，计算字符总长度。如有超过 20 字符（包含英文与符号）的，**强制强力压缩缩写**，否则不得进入生成队列。
2. **格式扫描**：检查所有的 `预期结果`，是否每一条都已经原原本本带上了 `[UI反馈]`、`[状态回写]`、`[审计记录]` 这三个前缀标签。缺一个都必须打回重写。
3. **覆盖率雷达 (Coverage Radar)**：强制对比“原子约束清单”，检查是否有任何文本中提到的“显性数据字段”（如：体积、物理属性相加）或“微小交互（如：弹窗 Cancel）”被漏掉。如果漏掉，立即补齐。
4. **安全与边缘扫盲**：翻阅前置梳理矩阵，强制核查是否遗漏了以下三种必考场景：【表单全部置空/未选的强阻断拦截】、【网络断崖式超时/并发双击的 L3 防抖幂等】、【业务逻辑中 AND/OR 条件的部分匹配降级流转】。若缺少，必须在此处主动补齐用例。

### 第三步：生成 JSON 并执行导出

> ⚠️ **字段 Key 硬性约束**：只允许使用下表中文字段名，禁止英文、禁止同义替换。

| 字段名 | 必填 | 合法值 |
|---|---|---|
| `用例编号` | ✅ | `TS-CA-001`（三位数字，连续递增） |
| `所属模块` | ✅ | 模块名，不加数字序号前缀 |
| `用例标题` | ✅ | 描述测试目的，20字以内 |
| `测试类型` | ✅ | `功能测试` / `接口测试` / `性能测试` / `安全测试` / `易用性测试` / `边界测试` / `异常测试` |
| `前置条件` | ✅ | 执行前系统满足的状态 |
| `测试步骤` | ✅ | 用 `\n` 分隔，格式："1. 操作\n2. 操作" |
| `预期结果` | ✅ | 必含：UI反馈 + 状态回写 + 审计记录 |
| `优先级` | ✅ | `P0` / `P1` / `P2` |
| `备注` | 选填 | 意图解释 / 风险补充 |
| `系统补录` | 🚫 | **系统自动生成字段**（用于捕获 AI 输出的非标字段，AI 无需主动填写） |

> **数据安全声明**：导出的 Excel 包含第 10 列 `系统补录`，用于自动捕获 AI 输出中所有未定义的额外字段，确保 100% 数据不丢失。

1. **输出 JSON（缓冲区模式 - 稳定性方案）**：
    - **策略**：为了 100% 规避“参数缺失”和“未读先写”以及“中文字符编码”报错，统一使用固定缓冲区文件：`skills/doublehit-case-skill/output/cases_buffer.json`。
    - **JSON 质量红线**：
        1. **绝对禁标（ASCII Only for Structure）**：不仅是引号，所有的冒号 `:`、逗号 `,`、中括号 `[]` 必须是半角。严禁在文案中使用全角引号封装关键信息，内部引号一律使用 `\"`。
        2. **序列化思维**：禁止手动拼接字符串，生成 JSON 时必须在内部逻辑中先将其作为 Data Object 处理，再一键转录。
        3. **工具选择**：严禁在 Windows 下使用 `echo` 写入长 JSON，必须调用系统的 `Write` 工具以确保 UTF-8 编码不乱码。
    - **标准执行流**：

        1. **Read**：先执行一次 `view_file` 读一下该文件（即使内容是 `[]`），以获得系统的“写入许可”。
        2. **Write**：将新生成的 JSON 内容覆盖写入此文件。
        3. **Verify**：**强制执行一次** `python -c "import json; json.load(open('skills/doublehit-case-skill/output/cases_buffer.json', encoding='utf-8'))"` 校验合法性。
        4. **Export**：校验通过后再调用脚本导出，失败则立即原地修复 JSON。
        5. **Clean**：任务结束后，将该文件重置为 `[]`。

2. **自动化导出命令适配**：

## 4. 使用场景
- **即时需求设计**：粘贴 PRD 片段，自动识别输入类型 + 选方法 + 生成。
- **模糊需求处理**：先追问关键信息，再生成，确保质量。
- **深度扩展探测**：用户要求"向外延伸"或"覆盖边缘场景"时的专项设计。
- **断点续写与增量生成（分段执行建议）**：
  - 如果用例总数预计超过 15 条，**强烈建议分批次生成**（如每 10 条一批）。
  - 先生成第一批并使用 `cat` 写入，待用户输入“继续”后再生成下一批并使用 `append` 模式追加。
  - 这样可以有效防止因单次输出过长导致的平台超时或 Token 溢出。
